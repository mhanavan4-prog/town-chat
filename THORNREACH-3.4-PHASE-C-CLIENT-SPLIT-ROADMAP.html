The server is now a set of injected modules wired together in one entrypoint. The client is still one file, one closure, 21k lines. Everything the earlier Monolith Split roadmap said about the client being the “Hard” half is true — and the reason is a single structural fact.

The one fact that changes everything: client.js is a single IIFE

The whole file is wrapped in one immediately-invoked function:

(function () {
  "use strict";
  const canvas = document.getElementById('game');
  let W = 0, H = 0;
  // … ~40 top-level state bindings: ws, world, players, me, walls,
  //    currentRoom, world2, DUNGEON_WORLD, messagesByRoom, … …
  // … 612 top-level function declarations …
  requestAnimationFrame(loop);   // ← the single boot call, at the very end
})();

Unlike server.js (CommonJS, real require/module.exports seams), client.js deliberately exposes nothing to global scope — it only touches the DOM and the one THREE global from the CDN. That is good hygiene, and it is exactly what makes the split hard:

Everything lives in one shared closure. All 612 functions and ~40 state variables (players, world, me, walls, ws…) see each other for free today. Split the file into naive ordered <script> tags and each file becomes its own scope — the cross-references silently break.

⚠ Amendment to the earlier roadmap

The Monolith Split doc floated “Option A — ordered <script> split, sharing the same global scope, exactly like the vendored libs.” That works for three.min.js because it assigns to window (THREE). client.js does the opposite — nothing is global. So an ordered-script split is not free: you would first have to route every cross-file symbol through an explicit shared namespace. Given the IIFE, a bundler is now the cleaner fit (below).

The strategy decision (make this once, up front)

ApproachHow state is sharedVerdict
A · Namespace-object ordered scripts First file establishes const TR = (window.__TR ||= {}); every fragment is its own IIFE reading/writing TR.*. No build step. Works, but you hand-convert ~40 state bindings + hundreds of cross-calls to TR., and manage N-script load order in index.html. Lots of mechanical churn, easy to miss a reference.
B · esbuild bundle of real ES modules recommended Author import/export modules under client/; esbuild bundles them back into one IIFE-wrapped public/client.js. Best fit. The code is already one closure, so a bundle reproduces identical runtime semantics. index.html still loads one <script>. Type-checking and a missing-import become hard errors instead of silent breaks.
C · Drop the IIFE → true globals Delete the wrapper; everything becomes a real global. Not recommended. Pollutes window, risks collisions (the file has ~40 short names like W, me, world), and changes strict-mode behavior. Mechanically simple, semantically dangerous.

Recommendation — go with B (esbuild).

The IIFE that makes an ordered-script split painful is what makes a bundler natural: bundling ES modules produces the same single-closure output the browser already runs, so the runtime contract doesn’t change. You get navigable source, per-module // @ts-check, and the compiler catches every cross-module reference you move. Start with a bundler; you never need to graduate away from it.

Prerequisite — the client’s seatbelt (before any slice moves)

Phase A/B were safe because the __testHooks surface let the unit suite validate every extraction. The client has no such harness — its only coverage is the browser smoke test (join → per-class UIs, 37 checks) plus check:parity. So the client split leans on a different seatbelt, and it must be confirmed green first:

  • Browser smoke test (test/smoke.browser.mjs) — the load-crash canary. It boots the real client headless; a broken bundle fails the join. This is the primary net.
  • Client error telemetry (Tier 3.1: window.onerror/onunhandledrejection → /client-error) — so a runtime break in a moved module surfaces instead of running silently.
  • Typecheck (tsc --noEmit) — add client/ to tsconfig.json include and grow // @ts-check per module; a missed import is then a compile error.
  • Parity (check:parity) — still guards the two mobile www/client.js copies, now against the built bundle.

Prove the pipeline before the payload: stand up esbuild + a build:client script and move one trivial leaf end-to-end (see Milestone 1) before touching real subsystems. Wire the build/parity/sync changes once, on throwaway-small code.

The recipe (every client slice — same spirit as A/B)

1. Pick a leaf block (few things call into it): audio, shop, arcade…
2. Move it verbatim into client/<name>.js.
     - `import { players, world, send, … } from './core.js'`  (what it READS)
     - `export function renderNpcShop(){…}`                    (what others CALL)
3. In the remaining client, import those functions where they were used.
4. npm run build:client       # esbuild → single public/client.js
5. npm run sync:mobile        # copy the BUILT bundle to the two www/ copies
6. npm run test:ci            # build → lint → typecheck → parity → unit → smoke
       green → PR + merge.  red → revert, diagnose.
7. One slice per PR. Move first, refactor later.

What has to change in tooling (the client’s __testHooks analog)

  • New: build:client — esbuild client/loop.js --bundle --format=iife --outfile=public/client.js (+ a sourcemap). This becomes the first step of test:ci, before parity/smoke, so a broken build fails CI loudly.
  • sync:mobile + check:parity now operate on the built bundle, not source. They’re already hardcoded to public/client.js — which is now the build output — so they keep working if the build runs first. Order matters: build → sync → parity.
  • index.html is unchanged — still one <script src="client.js"> after the vendored libs. This is the whole payoff of the bundler path: no N-script load-order juggling, no route-stub gymnastics.
  • Smoke test serves client.js — point it at the built bundle, or add a pretest build step so the file always exists before the browser boots.
  • tsconfig.json — add client/** to include; add // @ts-check per module as you go. Client type coverage grows one slice at a time, the same way the server modules did.

Milestones (stop-anywhere)

1 · Tooling proof — esbuild + build:client, move ONE leaf (audio.js), full gate green 2 · Leaf UIs — shop, arcade, notify, countermeasures 3 · Systems — inventory, hud, chat, input, combat-fx 4 · The bulk — scene + world-* builders, creatures, props, player-visuals 5 · core + net + loop last — the shared-state spine and the boot call

Each milestone leaves the game fully working and green. Milestone 1 is the real gate: it proves the pipeline on throwaway-small code, so every slice after it is routine. After Phase C, public/client.js is a build artifact, the source is a navigable client/ tree, and the monolith era is over.

Risks / gotchas

  • IIFE → module scope (the big one) — every cross-file symbol must become an explicit import/export. Good news: with the bundler + typecheck, a missed reference is a hard error at build time, not a silent runtime break.
  • Hoisting no longer saves you — today’s 612 functions are hoisted in one scope, so call-order never mattered. Across modules it does: the first few slices will surface more cross-references than you’d guess. That’s the coupling becoming visible — let it.
  • The build is now load-bearing — a broken build:client means no client at all. Keep it as the first step of test:ci so CI catches it before anything else runs.
  • Keep shared state in core; never import back from features — core exports state/helpers; feature modules import from core. If two features reference each other, route through core or pass a callback — don’t create a cycle.
  • Mobile — the two www/client.js are the built bundle; never hand-edit; sync:mobile after every build, then npx cap sync + rebuild.
  • Never split + refactor in one step — move verbatim with imports wired, get green, improve separately. Same rule that kept Phase A/B clean.
